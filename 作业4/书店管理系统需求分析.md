# 书店管理系统需求分析报告

## 项目概述

本系统是为书店提供的一套完整的书店管理系统，主要目标是方便顾客从网上购买书籍，同时可以让书店在网上从书籍批发商处进货，省去复杂的沟通，线下交流的过程，提供一个可靠方便的网上解决方案。
 

## 功能性需求

本系统的功能性需求主要包括顾客、店主和供货商三个方面的需求。

### 系统用况描述

| 用况名称 | 购买书籍                            |
| ---- | --------------------------------- |
| 参与者  | 顾客                                |
| 描述   | 顾客通过本系统查询找到自己想买的书籍，并创建、填写订单 |
| 前置条件 | 顾客已登陆系统                          |
| 后置条件 | 完成订单填写，记录进后台                  |
| 基本路径 | 顾客查询书籍=>找到对应书籍=>进入订单填写界面=>填写订单详情=>支付 |

| 用况名称 | 订单管理                            |
| ---- | --------------------------------- |
| 参与者  | 顾客                                |
| 描述   | 顾客可以管理已经创建的订单，并可在特定情况下取消订单 |
| 前置条件 | 顾客已登录系统             |
| 后置条件 | 顾客知晓已创建的订单信息，并完成相关操作                     |
| 基本路径 | 顾客选择订单管理=>选择要查询的订单=>进行后续操作（仅查看或是取消订单） |

| 用况名称 | 退货                         |
| ---- | ------------------------------ |
| 参与者  | 顾客                             |
| 描述   | 顾客在购买书籍之后，对书籍不满意，可以选择退货 |
| 前置条件 | 顾客成功购买了书籍，且登录系统                       |
| 后置条件 | 顾客完成退货，将书籍归还书店，并得到退款                |
| 基本路径 | 顾客选择退货申请=>填写退货申请单=>等待店主确认=>书籍寄还书店，得到退款 |

| 用况名称 | 评价                        |
| ---- | ------------------------------ |
| 参与者  | 顾客                             |
| 描述   | 顾客在购买书籍之后，可以在书籍信息之下填写评价 |
| 前置条件 | 顾客成功购买了书籍，且登录系统           |
| 后置条件 | 完成书籍评论，相关信息记录进后台，其他用户可以看到   |
| 基本路径 | 顾客选择订单管理=>选择已完成的订单=>选择填写评价=>进行评价 |

| 用况名称 | 书籍管理    |
| ---- | ------------------------------ |
| 参与者  | 店长                             |
| 描述   | 店长可以管理系统内的书籍信息，包括添加，删除，修改等操作 |
| 前置条件 | 店长登录系统          |
| 后置条件 | 完成书籍信息的修改，并记录进后台              |
| 基本路径 | 店长选择要进行的操作（增/删/改）=>进行进一步的详细操作 |

| 用况名称 | 顾客订单管理 |
| ---- | ------------------------------ |
| 参与者  | 店长                           |
| 描述   | 店长可以查看用户创建的订单，以便发货，或者在货物不足时取消订单 |
| 前置条件 | 店长登录系统  |
| 后置条件 | 店长得到相关订单信息，或是成功取消订单，通知用户并退款     |
| 基本路径 | 店长选择订单管理=>选择要查询的订单=>进行进一步操作 |

| 用况名称 | 进货管理 |
| ---- | ---------------------- |
| 参与者  | 店长                    |
| 描述   | 店长可以发布进货请求给供货商，或是管理已发布的请求           |
| 前置条件 | 店长登录系统               |
| 后置条件 | 进货请求创建完成，供货商收到信息，进行回应              |
| 基本路径 | 店长选择进货管理=>创建进货请求=>填写细节=>供货商收到请求 |

| 用况名称 | 用户管理                       |
| ---- | ------------------------ |
| 参与者  | 店长                       |
| 描述   | 店长作为本系统的管理者，可以查看所有用户的信息，并可以删除用户|
| 前置条件 | 店长登录系统 |
| 后置条件 | 店长得到用户信息，或是删除了某个用户                |
| 基本路径 | 店长选择用户管理=>选择要查看信息的用户=>可以选择删除用户 |

| 用况名称 | 批发书籍 |
| ---- | -------------------------- |
| 参与者  | 供货商                       |
| 描述   | 供货商可以将书籍批发给书店，需要先进行进货请求的相关操作 |
| 前置条件 | 供货商登录系统，且店长已发布进货请求 |
| 后置条件 | 供货商对请求做出回应（接受/拒绝/竞价），若成功接受请求，则将书籍交付于书店，同时收到款项                  |
| 基本路径 | 供货商选择进货请求查看=>进行回应=>若请求成立，则将书籍交付于书店，同时收到款项 |

## 非功能性需求

### 界面美观，方便易用

界面应当满足人的审美，且操作简便，功能清晰，容易使用。

### 响应速度快

系统延迟要尽可能小，使得用户在操作之后能在最短时间内得到回应。

### 数据可靠性

系统中的数据应安全可靠，应当做好备份工作，使得用户的信息、书籍的信息、订单的信息等不会丢失。

### 健壮性

系统应当具有一定的健壮性，应尽可能不陷入故障
